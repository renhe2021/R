## 修复：LLM API 请求频率超限导致投委会辩论全部失败

### 问题
运行 Pipeline Stage 7d（投委会辩论）时，12 个 Agent × 8 只股票 = 96 个 LLM 请求同时并发打到 `dev.fit-ai.woa.com` 代理 API，触发频率限制（error code 1620867020：`请求频率超出限制`）。所有模型（claude-sonnet-4-5、gemini-3.1-pro、gpt-5、deepseek-v3.2）全部失败，所有 Agent 降级为 HOLD (10%) fallback。

### 同时发现的另一个 Bug
`gpt-5` 模型降级时报错 `AsyncCompletions.create() got an unexpected keyword argument 'max_completion_tokens'`。原因：`_adapt_kwargs_for_model()` 把 `max_tokens` 重命名为 `max_completion_tokens`，但代理 API 不认识该参数。

### 修复方案

#### 1. 三层并发控制（`llm.py`）
- **Semaphore（全局）**：`LLM_MAX_CONCURRENCY = 3`，限制最大同时 3 个 LLM 请求在飞
- **请求间隔（全局）**：`LLM_MIN_INTERVAL_MS = 200ms`，两次请求之间最少间隔 200ms
- **rate limit 检测增强**：HTTP 400 中含"频率超出限制"时自动重试（等 3×attempt 秒），而非直接跳到下一模型

#### 2. 股票级并发限制（`unified_pipeline.py`）
- Stage 7d（投委会辩论）：最多 2 只股票同时辩论（原来 8 只同时）
- Stage 7e（LLM 深度分析）：最多 2 只股票同时分析（原来 8 只同时）

#### 3. Agent 级并发限制（`debate.py`）
- 每只股票的 12 个 Agent 不再全部同时启动
- `agent_semaphore = Semaphore(3)`：每只股票最多 3 个 Agent 同时调用 LLM

#### 4. GPT-5 参数修复（`llm.py`）
- `_adapt_kwargs_for_model()`：不再把 `max_tokens` 改名为 `max_completion_tokens`
- 代理 API 统一使用 `max_tokens`，只移除 GPT-5 不支持的 `temperature`/`top_p`

### 并发效果对比
| 场景 | 修复前 | 修复后 |
|------|--------|--------|
| 最大瞬时 LLM 请求 | 96（12 agent × 8 stock） | 3（全局 Semaphore） |
| 股票级并发 | 8 只同时 | 2 只同时 |
| Agent 级并发 | 12 个同时 | 3 个同时 |
| 请求间隔 | 0ms | ≥200ms |

### 修改文件
- `backend/app/agent/llm.py` — 添加 Semaphore + 请求间隔 + rate limit 检测 + GPT-5 修复
- `backend/app/agent/unified_pipeline.py` — Stage 7d/7e 股票级并发限制
- `backend/app/agent/committee/debate.py` — Agent 级并发限制

## 新功能：Pipeline 数据质量检查点 (Data Quality Checkpoint)

### 需求
用户希望 Pipeline 不要一步到底，在数据采集（Stage 2）完成后暂停，让用户能够检查数据质量、覆盖率、关键指标正确性等，确认无误后再继续后续分析。

### 设计
使用已有的 SSE Pause-Resume 模式（与 `data_source_required` 相同）：

1. Pipeline Stage 2 完成后，构建详细的数据质量报告
2. 缓存 Pipeline 状态到 `_checkpoint_cache`（class-level dict）
3. 发射 `checkpoint` SSE 事件 → Pipeline `return`（暂停）
4. Frontend 弹出模态框，显示逐股数据详情
5. 用户点击"继续" → Frontend 重新提交 Pipeline 请求，带 `resumeFrom: 3` + `resumeRunId`
6. Pipeline 从缓存恢复状态，从 Stage 3 继续

### 数据质量报告内容（`_build_data_checkpoint()`）
- **Summary**: 总股票数、获取成功数、已淘汰数、平均核心覆盖率、数据源、策略
- **Per-stock details**:
  - 状态（ok/warning/poor/eliminated）
  - 核心/扩展/历史 覆盖率百分比
  - 关键指标（PE、PB、ROE、D/E、市值、EPS、营收、FCF 等）
  - Stage 3 门槛淘汰预览（该股票会不会通过 Stage 3？）
  - 缺失核心字段列表
- **Warnings**: 数据获取失败、低覆盖率、Stage 3 淘汰风险

### 修改文件
- `backend/app/agent/unified_pipeline.py` — `run()` 添加 `resume_from`/`resume_run_id` 参数；`_checkpoint_cache`；`_build_data_checkpoint()` 方法
- `backend/app/api/agent_routes.py` — `PipelineRequest` 添加 `resumeFrom`/`resumeRunId` 字段；传递给 `pipeline.run()`

## 修复：Bloomberg 连接风暴（每次 Stage 2 反复连接断开 16+ 次）

### 问题
用户运行 Pipeline 时日志显示 Bloomberg Terminal 在 Stage 2 数据采集期间反复连接/断开约 16 次（10 秒内）。

### 根因分析
1. `_stage2_data_acquisition()` → `_fetch_single()` 对每只股票调用 `get_data_provider("bloomberg")`
2. 每次 `get_data_provider()` 创建新的 `CachingProvider(BloombergProvider())` 实例
3. 每个 `BloombergProvider` 实例创建独立的 blpapi session（开连接）
4. 临时 provider 被 GC 时 `__del__` 调用 `close()`（关连接，打日志"连接已关闭"）
5. 同时 `probe_bloomberg_first()` 也创建独立的 `BloombergProvider` 实例（又一个 session）
6. `tools.py` 的 `scan_history()` 直接 `BloombergProvider()` 绕过 factory（又一个 session）

### 修复方案（4 层防护）

#### 1. Provider 单例模式（`factory.py`）
- 新增 `_provider_instances` dict + `_provider_lock` 线程锁
- `_get_or_create_provider(key, factory_fn)` — 线程安全的单例获取
- `get_data_provider()` 所有分支改为走单例路径：同一 `provider_name` 只创建一次
- `clear_provider_cache()` — 显式清理 provider（关闭连接）
- `_wrap_with_cache()` 增加 `isinstance(CachingProvider)` 判断防重复包装

#### 2. Stage 2 共享 provider（`unified_pipeline.py`）
- provider 在循环外创建一次 `shared_provider = get_data_provider(chosen_source)`
- `_fetch_single()` 通过闭包复用 `shared_provider`，不再每只股票创建新实例

#### 3. Bloomberg `__del__` 静默化（`bloomberg.py`）
- `close()` 改为只在显式调用时打日志："连接已关闭（显式调用）"
- `__del__` 改为静默关闭 session，不打日志，避免连接风暴假象
- 单例模式下 `BloombergProvider` 被 `_provider_instances` 持有，不会被 GC

#### 4. 消灭绕过 factory 的直接构造
- `tools.py` `scan_history()`：`BloombergProvider()` → `get_data_provider("bloomberg")`（走单例）
- `factory.py` `get_all_available_providers()`：直接构造 → `get_data_provider("bloomberg")`
- `probe_bloomberg_first()`：直接构造 → `get_data_provider("bloomberg", **kwargs)`（走单例）

### 效果对比
| 场景 | 修复前 | 修复后 |
|------|--------|--------|
| Stage 2 Bloomberg session 数 | N（每只股票 1 个） | 1（共享单例） |
| probe 额外 session | 1 | 0（复用单例） |
| scan_history 额外 session | 1 | 0（复用单例） |
| GC 触发的"连接已关闭"日志 | N 次 | 0 次（静默化） |

### 修改文件
- `src/data_providers/factory.py` — 单例模式 + get_data_provider 重构 + probe 重构
- `src/data_providers/bloomberg.py` — __del__ 静默化 + close() 显式标记
- `backend/app/agent/unified_pipeline.py` — Stage 2 shared_provider
- `backend/app/agent/tools.py` — scan_history 走 factory 单例

## 修复：Bloomberg 并发数据串线（AAPL 数据为空 + 股票名称张冠李戴）

### 问题
Checkpoint 截图显示：
- AAPL 核心覆盖率 0%，所有指标为空
- MSFT 名字显示为 "PepsiCo Inc"，GOOGL 显示 "NVIDIA Corp"，AMZN 显示 "Tesla Inc"
- 数据张冠李戴，股票之间的数据被交叉分配

### 根因
上一个修复（Provider 单例）让所有股票共享同一个 `BloombergProvider` 实例（同一个 blpapi session），但 `_stage2_data_acquisition()` 通过 `run_in_executor` 并行获取多只股票。

blpapi session 的 `sendRequest()` / `nextEvent()` **不支持多线程并发**——Thread A 发送 AAPL 请求后，Thread B 的 `nextEvent()` 可能拿到 AAPL 的响应，导致数据串线。

### 修复
- `bloomberg.py`: `__init__` 添加 `self._lock = threading.Lock()`
- `fetch()` 改为 `with self._lock: return self._fetch_locked(symbol)`
- 确保同一时刻只有一个线程在执行 BDP/BDH 请求
- 性能影响：Bloomberg fetch 变为串行，但 CachingProvider 磁盘缓存保证第二次运行无影响

### 附加修复：CachingProvider 脏数据保护
- `cache.py` `fetch()`: 缓存前检查核心字段数量，< 3 个核心字段（price/pe/eps/market_cap/revenue/roe/pb/name）则跳过缓存，防止保存串线脏数据
- `cache.py` `_cache_path()`: `safe_symbol` 现在用 `re.sub(r'[/\.\s]+', '_', ...)` 替换所有不安全字符（包括空格），文件名从 `AAPL US Equity_bloomberg_...` 变为 `AAPL_US_Equity_bloomberg_...`
- **需要手动清除旧缓存**：`data/cache/` 目录下的 .json 文件可能是之前并发串线时的脏数据，需要全部删除后重新运行

## 缓存策略简化：关闭磁盘缓存 + 基准数据内存缓存

### 决策背景
用户反馈 AAPL 数据持续为空，根因是磁盘缓存保存了并发串线时的脏数据。用户建议"数据量不大可以每次都重新下载"。

### 分析
- 每只股票 Bloomberg fetch = 4 次 API 调用（1 BDP 47字段 + 1 BDH 8字段×10年 + 2 基准 BDP），~5-6秒/股
- 30只串行 ≈ 2.5-3 分钟，数据量不大
- Pipeline 内同次运行的数据复用（Module A: data_map 传递 Stage 4/5/6）已有效，不依赖磁盘缓存
- 磁盘缓存 JSON 方案在并发场景下脆弱（已被证实）

### 改动
1. **`factory.py`**: `_CACHE_ENABLED = False`（默认关闭磁盘缓存）
   - Pipeline 每次运行都从 Bloomberg/yfinance 重新获取，保证数据新鲜
   - data_map 内存传递依然有效（Stage 4/5/6 不重复 fetch）
   - 保留 CachingProvider 代码和 `set_cache_enabled()` API，用户可按需开启
2. **`bloomberg.py`**: 基准数据内存缓存（class-level `_benchmark_cache`）
   - SPX PE 和国债利率一天只变一次
   - 30只股票共享同一份基准数据，从 60 次查询降到 2 次
   - 用日期作为缓存 key，跨日自动失效

### 修改文件
- `src/data_providers/factory.py` — `_CACHE_ENABLED = False`
- `src/data_providers/bloomberg.py` — `_benchmark_cache` + `_fetch_market_benchmarks()` 改用缓存
